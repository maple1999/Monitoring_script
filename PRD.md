# PRD：每日 CV 机会情报 Digest（比赛/活动/实习）自动化脚本

版本：V1.0（基线锁定于 2026-03-04）
面向：研二计算机视觉方向（CV / 多模态 / AIGC / VLM / LLM）
部署：自有服务器 + 定时任务（08:45 Asia/Singapore）
投递：SMTP 邮件（发件 163 邮箱）
每日产出：3 条（比赛 1 + 活动 1 + 实习 1），按新增优先排序，必要时存量补位

------

## 1. 背景与目标

### 1.1 背景

用户希望自动化获取对研二 CV 研究与职业发展有帮助的信息，包括：

- Kaggle 竞赛
- 国内大厂/知名互联网公司举办的算法/开发/AI 竞赛与活动（含 AICode 等）
- 国内大厂/相关公司实习机会（偏 CV/多模态/AIGC/VLM/LLM，兼顾 AI/算法/大模型产品经理）

由于不同站点结构差异大、规则难以完全固定，系统允许“站点白名单 + 关键词搜索聚合”的策略，并引入 LLM 进行**总结评价**（V1 不负责抽取字段的主链路）。

### 1.2 产品目标

- 每天 08:45 自动发送一封邮件，仅包含**最值得看的 3 条机会**（比赛/活动/实习各 1）
- 每条机会提供**结构化关键信息 + LLM 难点评估与匹配度**
- 支持“新增 vs 昨日已存在”标识，优先推送新增
- 比赛与活动提供**临近截止提醒（≤14 天）**
- 系统出错/抓取异常/LLM 失效/内容不合理时，必须发送**告警邮件**

### 1.3 非目标（V1 不做）

- 不做完整的“任意网页字段抽取全自动化”（LLM 不承担主链路结构化抽取）
- 不做 UI 管理后台（仅 config 文件配置）
- 不做付费邮件服务依赖（优先 SMTP 免费方案）
- 不做大规模爬虫（每日候选上限严格控制）

------

## 2. 用户画像与核心场景

### 2.1 用户画像

- 研二 CV 方向，关注竞赛与实习对科研/简历提升的价值
- 希望减少信息噪声，快速看到“最相关、规模更大、时效性更强”的机会
- 能接受少量配置（关键词/白名单/SMTP），但希望脚本自动运行稳定

### 2.2 核心用户故事（User Stories）

1. 我每天早上收到一封邮件，里面只有 3 条最相关机会，省去刷网站。
2. 我希望每条机会都能看到：关键信息 + 难点分析 + 对我方向的匹配度与理由。
3. 我希望新增机会优先，截止临近的比赛/活动有提醒。
4. 如果脚本出错或 LLM 不可用，我也要收到邮件提醒，避免“静默失败”。

------

## 3. 范围与信息类型定义

### 3.1 类型（Type）

- **Contest（比赛）**：Kaggle 竞赛 + 国内大厂/国家级竞赛（算法/开发/AI 方向）
- **Activity（活动）**：大厂/机构主办的 AI/开发/算法活动（训练营、挑战赛、Hackathon、公开赛、开发者大赛等）
- **Internship（实习）**：技术实习（研究/算法/产品），方向优先 CV/多模态/AIGC/VLM/LLM

### 3.2 每日条数

- 每日仅 3 条：
  - 比赛 Top1
  - 活动 Top1
  - 实习 Top1

## 4. 数据源与采集策略

### 4.1 总体策略：站点白名单 + 关键词检索

- 系统维护一组**站点白名单（domains）**，按来源类型分类：
  - Kaggle（固定域名）
  - 公司官网（竞赛/活动/开发者/新闻/社区）
  - 公司招聘官网（实习/校招）
  - 国家级竞赛官网/主办方平台页（固定域名）
- 在白名单站点内定期通过关键词检索/站内搜索/列表页遍历获取候选链接
- 允许少量“落地页/镜像/专题页”域名，但必须满足“来源可信 + 内容可追溯”

### 4.2 公司白名单（已确定）

腾讯、阿里巴巴、字节跳动、美团、京东、拼多多、百度、快手、小米、携程、蚂蚁集团、网易、滴滴、知乎、贝壳、360、小红书、新浪微博、SHEIN、得物、爱奇艺、米哈游、华为、科大讯飞、寒武纪、浪潮信息、地平线机器人、智谱AI、旷视科技、商汤科技、沐曦、联想、比亚迪、大疆创新、深度求索（DeepSeek）

> 注：PRD 规定“脚本自行整理白名单站点”，实现阶段会落到 `domains_allowlist.yaml`，可手工补充/修正。

### 4.3 国家级竞赛（已举例，允许扩展）

- 研究生数学建模竞赛
- 研究生人工智能创新赛
  （后续可在配置中追加关键词与官网域名）

### 4.4 候选抓取上限（每日）

- 比赛候选上限：15
- 活动候选上限：15
- 实习候选上限：30

> 上限含“新抓取 + 存量候选筛选池”，最终只发 Top1。

## 5. 字段定义与输出规范

### 5.1 统一 Item 数据模型（最小必需字段）

- `item_id`：内部唯一（hash(source + url)）
- `category`：contest / activity / internship
- `title`：标题
- `title_en`（可选）：英文标题（若原文纯英文）
- `title_zh`（可选）：中文翻译（纯英文时必填）
- `url`：来源链接（必须可追溯）
- `source`：来源名（Kaggle / company_x / official_site 等）
- `company_or_org`：公司/机构（尽力解析；不确定则为空）
- `publish_time`（可选）：发布时间
- `deadline`（可选）：截止时间（比赛/活动）
- `time_range`（可选）：开始/结束/阶段信息（报名/初赛/复赛/决赛）
- `requirements`（可选）：参赛要求/岗位要求（尽力抽取）
- `location`（实习可选）：地点
- `work_mode`（实习必填）：online / offline / unspecified（规则：未说明默认 offline）
- `summary`：页面摘要（规则抽取或简单摘要）
- `summary_en`（可选）：英文摘要（若原文纯英文）
- `summary_zh`（可选）：中文翻译（纯英文时必填；仅翻译标题+摘要）
- `tags`：关键词标签（CV/多模态/AIGC/VLM/LLM 等）
- `first_seen_time`：首次发现时间（用于新增与存量有效期）
- `last_seen_time`：最近一次抓取时间
- `is_new`：是否新增（URL 首次出现=新增）
- `status`：active / expired / invalid

### 5.2 比赛（Contest）输出字段（邮件展示）

- 主题/标题（中英按规则展示）
- 链接
- 赛题/任务描述（从页面摘要与关键段落整理）
- 时间（开始/截止/阶段；若缺失写“未解析到/页面未提供”）
- （可选）要求（资格/提交/赛制/数据限制）
- **LLM 段落：难点评估 + 匹配度 + 评价 + 补充信息**

### 5.3 活动（Activity）输出字段（邮件展示）

- 主题/标题
- 链接
- 时间（报名/活动日/阶段；若缺失写清楚）
- 要求（参与方式/形式/提交/门槛）
- **LLM 段落：同上**

### 5.4 实习（Internship）输出字段（邮件展示）

- 职位标题（可含部门）
- 公司
- 链接
- 地点
- 线上/线下（未说明默认线下）
- （可选）截止/投递信息
- **LLM 段落：同上（但不需要 DDL 提醒）**

## 6. LLM 介入范围与输出规范（V1）

### 6.1 介入边界（V1）

- LLM **不承担主链路字段抽取**，仅对已抓取的页面文本/摘要进行：
  1. 难点评估（数据/工程/数学算法）
  2. 方向匹配度评分（0–5）与理由
  3. 规范化评价（2–3句，严谨，不比喻）
  4. “补充信息”：当 LLM 判断页面存在对用户显著有用的信息（例如奖励、评测指标、数据规模、提交限制、赛制阶段等），可补充；否则写“无”
- LLM 不能编造：缺失必须明确说明“页面未提供/未解析到”

### 6.2 固定输出模板（单段落）

**必须严格按以下顺序输出：**
**难点评估：**数据难点=…；工程难点=…；数学/算法难点=…；**匹配度=R/5**（理由：…）；**评价：**…（2–3句，逻辑严谨、无比喻）；**补充信息：**…（无则写“无”）。

### 6.3 语言与翻译

- 邮件中文为主
- 若标题+摘要为**纯英文**：
  - 标题展示：`[EN] title_en` + `[ZH] title_zh`
  - 摘要展示：`[EN] summary_en` + `[ZH] summary_zh`
  - 仅翻译标题+摘要；其余字段按原始页面可得信息展示（如要求/时间），不强制翻译全文
- 中英混合内容：不翻译，直接展示

------

## 7. 排序、评分与选取逻辑

### 7.1 总体规则

- 每类（比赛/活动/实习）先生成候选池（上限见 4.4）
- 对候选去重、过滤、评分
- 每类选 Top1 作为当日推送
- 邮件中三类条目按“新增优先 + 新增顺序”排序展示：
  - 新增的条目优先展示
  - 若均为新增或均非新增，则按 `first_seen_time` / `last_seen_time` 近者优先（实现时明确为配置项）

### 7.2 新增判定（已确定）

- **新增=官方 URL 首次出现**
- 记录 `first_seen_time`，后续同 URL 均为“昨日已存在/已存在”

### 7.3 存量补位（已确定）

当某一类当天找不到足够高质量候选时，允许从历史未推送但仍有效的存量中补位：

- 存量有效期：统一 `first_seen_time + N days`（N 可配置，默认 30）
- 若解析到 DDL 且 DDL 已过：直接失效（即使 N 未到）
- 存量补位仅用于“凑齐三类 Top1”，不会增加每日条数

### 7.4 赛事规模（Contest/Activity）排序信号（已选 B）

“规模大小”作为评分的一个因子，基于可获取信号做 proxy，优先级建议：

1. 主办方级别：国家级/官方平台 > 一线大厂（白名单）> 其他
2. 是否有奖金/奖项/奖池：有 > 无/未知
3. 是否有线下总决赛/宣讲/官方发布会：有 > 无/未知

> 注：V1 不要求精确参赛人数/曝光量；只用可稳定抽取的信号。

### 7.5 实习方向与岗位类型排序规则

- 方向关键词优先：CV > 多模态 > AIGC > VLM > LLM
- 岗位类型优先：研究型/算法型/AI产品（大模型/算法/AI产品经理）优先
- 若没有 AI/算法/大模型产品岗，可降级接收其他产品经理岗位（噪声更高但允许）
- 排除：测试、运维 等（关键词黑名单）

------

## 8. 截止提醒规则（比赛/活动）

### 8.1 临近截止提醒

- 仅对**比赛/活动**生效
- 若可解析 `deadline` 且 `deadline - today ≤ 14 天`：标记“临近截止”
- 实习不做临近提醒（即使有截止）

### 8.2 无 DDL 情况

- 若抓不到 deadline：不做临近提醒
- 但在条目字段中备注：“未解析到DDL/页面未提供”

## 9. 邮件产出规范

### 9.1 日报邮件

**主题建议：**
`Daily CV Digest - YYYY-MM-DD（比赛/活动/实习各1）`

**邮件结构：**

1. 顶部概览
   - 运行状态：成功 / 降级（例如 LLM 不可用但仍发）
   - 新增情况：比赛/活动/实习的新增/已存在标识
   - 提醒：若比赛/活动临近截止，顶部汇总提示
2. 三条正文（按新增优先排序）
   - 每条：类别标签 + 新增/已存在 + 标题（中英规则）+ 链接 + 关键字段 + LLM 段落
3. 运行信息（简短）
   - 候选抓取数量（比赛≤15/活动≤15/实习≤30）
   - 若某来源失败：列出失败来源（不展开长日志）

### 9.2 告警邮件（必须支持）

**触发：**脚本异常 / 抓取异常 / LLM 异常 / 内容异常（见 10）
**主题建议：**
`[ALERT] CV Digest Failed/Degraded - YYYY-MM-DD`

**内容必须包含：**

- 错误类型：Run / Crawl / LLM / Content
- 影响范围：比赛/活动/实习
- 关键错误信息（简化、可定位）
- Run ID / 日志位置（路径或标识）
- 是否仍发送日报：是/否
- 建议操作（例如：检查 SMTP 授权码、检查 LLM Key、检查站点结构变更）

------

## 10. 异常与质量闸门（Quality Gates）

### 10.1 异常类型与处理

1. **Run 级错误**：脚本崩溃、DB/Scheduler/SMTP 失败
   - 处理：立即尝试发送告警邮件；若 SMTP 也失败则落盘并在下次日报顶部提示“上次失败”
2. **Crawl 异常**：
   - 某类候选为 0（连续 ≥K 次，默认 K=2）
   - 候选大量为空/重复异常（如重复率>80%）
   - 处理：降级（启用存量补位），并发告警（或在日报标注“降级”）
3. **LLM 异常**：API 失效/超时/鉴权失败/返回空
   - 处理：降级策略：
     - 日报仍发送，但 LLM 段落使用“Fallback 模板”（基于规则/关键词生成简短说明，并注明“LLM不可用”）
     - 同时发告警邮件（或在日报顶部标红并单独告警，取决于配置）
4. **Content 异常（不合理产出）**：
   - LLM 输出不符合模板（缺字段/出现比喻/无意义文本）
   - 选出的比赛/活动 DDL 已过
   - 匹配度全部低于阈值仍被选中且没有存量补位理由
   - 处理：该条目剔除，改选下一名；若无可用则存量补位；同时告警

### 10.2 降级与兜底策略

- 优先保障“每天 08:45 能发出邮件”
- 在可控范围内牺牲“完美字段完整性”，但必须保持：
  - 链接可追溯
  - 不编造
  - 明确标注缺失/降级原因

## 12. 技术与系统流程（实现导向）

### 12.1 模块划分（建议）

1. Scheduler：定时触发（08:45）
2. Source Manager：站点白名单管理（自动生成 + 手工覆盖）
3. Collector：按类别抓取候选（上限控制）
4. Parser（规则抽取）：提取标题/时间/要求/摘要/地点/工作模式（尽力而为）
5. Normalizer：统一 Item schema
6. Deduplicator：URL 去重 + 标题相似去重（可选）
7. Scorer：匹配度初筛 + 规模信号 + DDL 紧迫度（比赛/活动）
8. Selector：Top1 + 新增优先 + 存量补位
9. LLM Evaluator：生成固定段落（含翻译策略）
10. Mail Renderer：HTML + text fallback
11. Mail Sender：163 SMTP
12. Logger & Store：SQLite/JSONL 记录 items、runs、send_log
13. Alert Manager：告警检测与邮件发送

### 12.2 一次运行的主流程

1. 读取 config + 环境变量
2. 抓取候选：contest≤15、activity≤15、internship≤30
3. 规则抽取字段 + 统一 schema
4. 去重（按 URL）+ 标注 is_new
5. 评分与排序（含规模 B）
6. 每类选 Top1；不足则按存量补位规则补齐
7. 对选中的 3 条生成 LLM 段落（若失败则 fallback）
8. 生成日报邮件并发送
9. 写入 run 日志与 send_log
10. 若触发异常/降级/不合理内容闸门：发送告警邮件（或在日报标红 + 单独告警，按 config）

------

## 13. 验收标准（Definition of Done）

### 13.1 功能验收

- 每天 08:45 自动发送日报邮件
- 邮件固定包含 3 条：比赛/活动/实习各 1
- 每条包含：
  - 标题、链接、关键字段（按类型要求）
  - 新增/已存在标识
  - LLM 固定段落（或明确 fallback 标注）
- 比赛/活动：若 DDL ≤14 天，必须有“临近截止”提示
- 实习：必须标注线上/线下（未说明默认线下）
- 纯英文条目：标题+摘要同时展示英文与中文译文

### 13.2 稳定性验收

- 任意以下情况必须发告警邮件：
  - 脚本异常退出
  - LLM API 不可用导致无法生成段落（即便日报降级发送）
  - 内容闸门命中（模板违规、DDL 过期仍推送等）
- 日志可定位：每次运行有 Run ID/时间戳/成功失败状态
